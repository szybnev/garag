## Трек A. RAG-система по выбранному корпусу с метриками, адаптацией и оптимизациями

### Цель
Спроектировать, создать и развернуть RAG-систему для ответов на вопросы по выбранному вами корпусу (любой домен). Система должна давать точные ответы с цитатами/ссылками, быть покрыта метриками и оптимизирована под приемлемые для домена нефункциональные требования.

В качестве данных можно использовать любые тексты: документы, статьи, спецификации, новости. Можно опционально добавить мультимодальность (OCR изображений, PDF парсинг...).

Соберите золотой набор из 50–200 query-relevant пар для объективной оценки ретривера и генерации.

### Функциональные требования

#### Поиск и ответ
- hybrid поиск с гибкой экспериментально настроенной комбинацией (RRF/MMR).
- Реранжирование топ-кандидатов кросс-энкодером (если улучшает качество и укладывается в НФТ).
- Генерация ответов с привязкой к источникам (список цитат с id/URL/фрагментом).
- Структурированный вывод со схемой {answer, citations[], confidence, used_chunks[]}.

#### Оценка и мониторинг
- Автооценка retrieve-части (Recall@k/nDCG/MAP на золотом наборе).
- Оценка ответов: LLM-as-judge на faithfulness/корректность + выборка ручной проверки.
- Трассировка, логи запросов, технические метрики - e2e latency + latency и ключевые системные метрики по компонентам пайплайна

### Нефункциональные требования

Сформулируйте НФТ под свою систему/домен и зафиксируйте в дизайн-документе: - **5 баллов**
- Время ответа
- Пропускная способность: не менее 
- Время индексации/обновления данных
- Масштабируемость
- Доля релевантных ответов (accuracy / precision@k / nDCG) по тестовому набору
- Интерпретируемость и цитируемость
- Целевые лимиты по использованию GPU/CPU/памяти на инстанс.
...

### Технические требования

#### Индексация/поиск - **10 баллов**
- Построить индекс в векторной БД (Qdrant/Milvus/Weaviate/PGVector на выбор). Настроить параметры индекса под оптимальное качество/скорость индексации и поиска - **3 балла**
- Выбрать оптимальный (качество/ресурсы) энкодер под вашу задачу, обосновать выбор - **2 балла**
- Реализовать и настроить hybrid поиск (sparse (BM25/TF-IDF) + dense (эмбеддинги)) - **2 балла**
- Улучшить ранжирование: тюнинг BM25 для вашего корпуса, RRF/MMR, кросс-энкодер. - **3 балла**

#### Чанкинг и загрузка - **3 балла**
- Выбрать и обосновать (теоретически или экспериментально) стратегию чанкинга под вашу задачу

#### [ОПЦИОНАЛЬНО] Адаптация моделей - **4 балла за любой из пунктов**
- Дообучить энкодер в contrastive режиме на парах (Q, pos/neg) из вашего корпуса.
- Адаптировать выбранную LLM в instruct-режиме с QLoRA для вашего домена.
- Попробовать улучшить качество генерации через DPO + QLoRA  или техники без RL

#### Инференс, деплой и оркестрация - **3 балла**
- Подберите оптимальные параметры генерации под задачу
- [ОПЦИОНАЛЬНО] Реализуйте хостинг LLM через vLLM - если есть ресурсы на развертывание

#### Инфраструктура - **5 баллов**
- API (REST/gRPC) + простой веб-интерфейс - **2 балла** 
- Деплой - облако/локально с Docker Compose - **2 балла**
- [ОПЦИОНАЛЬНО] Мониторинг/логирование/трейсинг - Prometheus+Grafana - **1 балл**

#### Безопасность - **5 баллов**
- Проведите тестирование чат-бота любым open-source инструментом (garak, giskard, promptfoo) - **3 балла**
- Установите guardrails (на выбор: llm guard, nemo guardrails и т.д.) - **2 балла**
